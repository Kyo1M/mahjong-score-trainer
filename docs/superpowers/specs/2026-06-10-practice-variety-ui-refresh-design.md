# 出題バリエーション拡充＋UI/動線刷新 設計

日付: 2026-06-10
ステータス: 承認待ち
前提: ランダム出題エンジン（2026-06-08 spec / PR #3, `feat/random-practice-engine`）の上に積む改善。

## 背景と課題

ユーザーから挙がった不満:

1. 問題タイトルに役名が入っており、答えのネタバレになっている（`question-factory.ts` の `title: ○○を含む手`）
2. 練習問題の幅が狭い。一気通貫や七対子など同じような役・同じような牌姿ばかり出る（アーキタイプ10種、各アーキタイプの形がほぼ固定。符も20/25/30/40しか出ない）
3. UI・移動の動線が洗練されていない

## 決定事項（ブレインストーミングでの合意）

- ビジュアル方向: **現行の暖色系（ベージュ×真鍮）を維持しつつフラットに洗練**（ダーク化・ミニマル白地案は不採用）
- 進行モデル: **エンドレス出題を維持して細部を磨く**（セッション型・10問区切りは不採用）
- 出題拡充: 役の種類を増やす ＋ 符ドリル（50〜70符）＋ 同一役でも形をランダム化、の3方向すべて
- 生成アーキテクチャ: **案1 = 汎用ブロックビルダー＋テーマドリル併存**（手書きアーキタイプ増産のみ・完全ランダム生成は不採用）
- 回答欄の並びを **符 → 翻 → 支払い** に変更（「30符4翻」の読み順）
- 役満生成・テーマ別成績分析・間違い振り返りリストは**スコープ外**（役満は従来どおり、分析系は今回は見送り）

## 設計 1: 出題エンジン

### 新モジュール `src/domain/hand-builder.ts`（ブロックビルダー）

完成形14枚を部品から組み立てる汎用ビルダー。

1. **ブロック抽選** — 雀頭1＋面子4。各面子は順子/刻子を重み付き抽選（順子寄り）。スート・数字はランダム、字牌刻子は低確率。`countsOk`（同一牌4枚以内）で矛盾形を排除
2. **待ち抽選** — 完成形から和了牌を選ぶ: 順子の端（両面）／中央(嵌張)／123・789 の 3・7（辺張）／雀頭（単騎）／刻子（シャンポン）。最終妥当性はエンジン（majiang-core）の `valid` 判定が担保
3. **副露** — 0〜2面子をポン/チー化。副露時は立直・一発を付けない（既存 `accept()` の検査と整合）
4. **状況抽選** — 親/子、ロン/ツモ、立直（門前のみ・確率）、一発（立直時のみ）、場風/自風（既存 `ctx()` の流儀を踏襲）
5. **ドラ 0〜2枚** — `safeDoraIndicator`（常にドラ0）を「手牌中にちょうどN枚当たる表示牌を選ぶ」に拡張。ドラを翻に足す練習を出題範囲に含める

役なし手・不形はエンジンが弾く → 棄却して再抽選（既存の400回リトライの枠内）。

### テーマドリル（アーキタイプとして併存）

ランダムでは出にくい・狙って練習させたいテーマは専用ビルダーで維持・追加:

- **符ドリル（新規）** — 幺九牌・字牌の暗刻2〜3個＋単騎/嵌張待ちで 50〜70符 を構成。翻は低め（役牌 or 立直程度）にして「符が点数を決める」問題にする
- **七対子・混一色・清一色・役牌（既存）** — 維持しつつ、スート・待ち・面子構成をブロックビルダーの部品でランダム化
- **制約付きビルダー** — 一気通貫（同一スート123/456/789を強制）、三色同順（同数字の順子を3スートに強制）などは専用アーキタイプではなく、ブロックビルダーへの制約注入として実装

### 出現分布と「同じ役ばかり」対策

- mix 時の抽選比率: ブロックビルダー約60% / テーマドリル約40%（テーマ内は均等）
- **バラエティガード（新規）** — 直近数問の主要役キーの組合せを保持し、同一構成が連続したら棄却して引き直す
- 難易度フィルタは従来どおり `classifyDifficulty` による事後判定。`accept()` の検査（valid・14枚整合・カタログ整合・副露と立直の矛盾・現実性）も従来どおり

### カタログ追加

- `yaku-map.ts`: **小三元・三色同刻** を `yakuCatalog` と `EXACT` に追加（ビルダーが偶発的に作っても棄却されないように）
- `question-factory.ts` の `CONFUSABLE` に対応エントリを追加（例: 小三元⇔役牌群、三色同刻⇔三色同順・対々和）

### ネタバレ除去

- `buildQuestion` の `title` から役名を排除。タイトルは状況表記のみ（例: 「子のロン和了」）。`prompt` は重複しないよう簡素化
- 静的問題集（`questions.ts`、テスト専用）はタイトルに役名が残るが、UIから参照されないため変更しない

## 設計 2: UI・動線

### 動線（エンドレス磨き込み）

- **タイトル行** — eyebrow「第N問」＋ h1「子のロン和了」＋ 右側に難易度チップと難易度セレクタ
- **難易度セレクタ** — 5ボタン横並びの `difficulty-bar` を廃止し、カード右上のコンパクトなセレクタへ
- **回答欄** — 符 → 翻 → 支払い の順（従来は 翻 → 符 → 支払い）
- **採点後** — (1) 解説パネルへスムーズスクロール (2) 画面下に固定バー（結果サマリ＋「次の問題へ」ボタン）を表示。解説を読み切らなくてもいつでも進める
- **Enterキー** — 未採点時は採点（`canSubmit` 時のみ）、採点後は次の問題へ。ボタンに `Enter` のヒント表記
- **ヘッダー** — 今セッションのミニ成績「8問・正解6」を常時表示
- **「終了して結果へ」** — ボタンからテキストリンクへ格下げ（誤クリック防止）。ヘッダーの「結果」ナビは残す

### ビジュアル（暖色系の洗練）

- `index.css`: 背景の radial-gradient 3枚を削減しフラットな暖色ベースに。カラートークン（`--bg` `--brass` 等）は維持して値のみ微調整
- `App.css`: カードの多層グラデーション→単色フラット面、`::before` 装飾ブロブ削除、角丸 28px→14px 前後、影を軽量化、ボタンのグラデーション→単色
- 見出しサイズ縮小（h1 最大 4.6rem → 実用サイズ、練習画面は ~2rem 程度）
- 正誤色・卓面の緑・牌タイルの見た目は維持

### コード構成

- `App.tsx`（815行）を分割: `src/pages/HomePage.tsx` / `PracticePage.tsx` / `GuidePage.tsx` / `ResultsPage.tsx`、共有UIは `src/components/` へ（`AnswerGroup`、`FeedbackPanel`、`ContextPanel` など）
- `App.tsx` はルーティング・セッション状態・ヘッダーのみ
- セッション保存（`sessionStorage`、`storageKey` v3）は形式変更なし

## エラー処理

- 生成失敗時は従来どおり: 第1パス（accept）400回 → 緩和パス400回 → 例外。ブロックビルダーの棄却率が上がっても、テーマドリルが混ざるため実用上の失敗率は従来同等を見込む
- バラエティガードは「棄却条件の追加」であり、リトライ上限内で満たせない場合はガードを無視して採用（無限ループ・例外増加を防ぐ）

## テスト

- **統計テスト（新規）** — seed 固定で N=200 問生成し検査:
  - 全問 `valid` かつ14枚整合（`countsOk`）
  - 出現役の種類数が閾値以上（mix で10種以上）
  - 50符以上の問題が一定数含まれる
  - 同一の主要役構成が連続しない（バラエティガードの検証）
- **符ドリル** — 生成問題の fu が 50〜70 に収まることを検査
- **既存テスト** — golden.test.ts・scoring.test.ts（静的問題集）・既存 generator.test.ts は維持。タイトル変更に伴う question-factory.test.ts の期待値を更新
- **UI** — Enterキー操作と固定バー表示は Testing Library でユニットテスト

## スコープ外

- 役満・数え役満の生成、裏ドラ・カンドラ・赤ドラ、本場・供託・包（従来どおり）
- セッション型進行（10問区切り）、テーマ別成績分析、間違い振り返りリスト
- ルール変更（STANDARD: 切り上げ満貫なし・30符4翻=7700 を維持）
